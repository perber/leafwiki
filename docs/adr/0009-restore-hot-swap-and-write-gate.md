# ADR-0009: Live restore hot-swaps only `AuthService`'s user store, gated by a blocking write-gate, with self-restart as a fallback

## Status

Accepted (2026-07-19).

## Context

The Full Backup (snapshot) feature ([ADR](../../plans/full-backup-restore.md) — see also the README) can produce a ZIP containing `root/`, `assets/`, `branding/`, `branding.json`, `schema.json`, and `users.db`, but had no restore path. Restoring while the server keeps serving requires swapping live data out from under a running process — the risky part is not the file swap itself (rename-aside/rename-in is a well-understood pattern, see `internal/backup`'s git-conflict handling for a similar "detect, don't guess" posture) but the two things that hold *open, stateful handles* onto that data: `AuthService`'s `*sql.DB` connection to `users.db`, and every in-flight HTTP request that might read or write during the swap window.

## Decision

**Only `AuthService.userService` is a live pointer swap; everything else is either a plain file swap or a resync.** `internal/links`, `internal/tags`, `internal/properties`, `internal/search` are derived indexes, not part of the snapshot ZIP at all (ADR-0001) — after the file swap, the existing resync machinery (ADR-0004) rebuilds them in place through their *already-open* handles, no reopening needed. `internal/branding`'s `BrandingService` caches config in memory but holds no DB handle — a new `Reload()` method re-reads it after the swap. `sessions.db` isn't in the snapshot and is never swapped; a restore instead calls `AuthService.InvalidateAllSessions()` (`DELETE FROM sessions`) so every session — including the admin's own — is invalidated, forcing re-login as a deliberate consequence of "the users.db underneath you just changed," not an oversight. This leaves `AuthService.userService` (backed by `users.db`, which *is* in the ZIP) as the one thing that structurally requires a live handle swap, via `ReplaceUserStore(storageDir)`: open a fresh `UserStore`/`UserService` against the swapped-in file, swap the pointer under a new `sync.RWMutex` that guards only that one field (not `sessionStore`, which is never swapped and needs no lock), close the old handle after releasing the lock.

**A blocking write-gate (503, not a request queue) stops new mutations for the swap's duration.** `internal/restore.WriteGate` is engaged right before the file rename step and disengaged once the whole sequence (swap, auth reopen, session invalidation, branding reload) has committed; `internal/http/middleware/maintenance.WriteGateMiddleware` returns `503` for any non-GET/HEAD/OPTIONS request while engaged (mirrors the GET/HEAD/OPTIONS exemption `security.CSRFMiddleware` already uses, plus an exemption for the `/api/admin/restore` path itself, since the trigger endpoint fires before the gate engages and the self-restart recovery endpoint must stay reachable *while* it's engaged). A queue (buffer writes, replay after the swap) was rejected: it would need to serialize arbitrary domain writes generically, across every `internal/wiki/<domain>`, for an operation the project's own principles frame as rare and admin-triggered — "minimal operational complexity" (root `CLAUDE.md`) favors the simpler failure mode of "briefly unavailable, then normal" over building a generic write-replay mechanism for a few seconds of downtime.

**Rollback keeps `.pre-restore-*` backup copies until the entire sequence succeeds, not just the file-rename step**, so a failure in *any* phase (swap, auth reopen, branding reload) can cleanly roll every phase back — including re-pointing `AuthService` back at the original `users.db` if it had already been swapped (a rollback that only restored files but left `AuthService`'s handle on the now-deleted, still-open new database would silently diverge from what's on disk). Only if rollback *itself* fails is the job marked `NeedsIntervention`: the write-gate stays engaged (fail closed — a half-restored instance should not accept further writes) and the only supported way out is `internal/restore.SelfRestart()`, which re-execs the process (`syscall.Exec` on Unix; spawn-then-exit on Windows, mirroring `cmd/leafwiki/signals_unix.go`/`signals_windows.go`'s existing OS split for process-lifecycle code). Self-restart is deliberately the fallback, not the primary mechanism: a hot-swap that succeeds (the common case) never pays the cost of a process restart at all; a fresh cold boot (`NewWiki()`) is simply the one thing guaranteed correct by construction when the in-memory/on-disk state has diverged in a way this code can no longer reason about.

**An offline CLI path (`leafwiki restore-snapshot <zip>`) reuses the same validate/extract/swap primitives** (`internal/restore.RestoreOffline`) with none of the live machinery — no write-gate, no `AuthService` reopen, no resync trigger — because the process isn't serving yet; the next `NewWiki()` cold boot picks up the restored files for free. This is the disaster-recovery / fresh-instance-migration path, distinct from the admin-UI hot-swap path, which only ever restores one of *this instance's own* already-produced snapshots (no arbitrary-ZIP-upload endpoint exists).

## Consequences

- `internal/restore` has zero HTTP/gin dependency (mirrors `internal/snapshot`); `internal/http/middleware/maintenance` is the only place that knows about gin, keeping the write-gate's core logic testable without spinning up a router.
- A request already in flight against the old `*UserService` when `ReplaceUserStore` swaps may see a transient "database is closed"-shaped failure in the sub-second window around the swap — accepted, not engineered around, since the write-gate already stopped new *mutating* traffic and a failed GET simply retries against the new store on its next call.
- Restore only ever brings back what a given snapshot actually captured — an item absent from the ZIP (e.g. an older snapshot taken before `branding.json` was added to the format, or a genuinely empty `assets/` at snapshot time) is left untouched on live disk rather than cleared, so a restore can never *delete* live content the snapshot simply didn't record.
- No snapshot-version compatibility matrix exists yet; a version mismatch between the snapshot and the running binary surfaces as a non-fatal warning on the restore job status, not a hard block. Revisit if cross-version restore ever produces a real incompatibility.
